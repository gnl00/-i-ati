# repositories

`repositories/` 面向领域实体，负责把多个 dao / mapper 组合成稳定的持久化边界。

适合放在这里的内容：
- 领域实体的读取与保存
- 多个 dao 的协调
- 需要事务保证的一致性写入
- 持久化层内的聚合规则，例如计数回写、关联记录同步

不应该放在这里的内容：
- 直接散落的原始 SQL
- UI / IPC / host 侧编排
- 跨多个大业务模块的流程控制

约束：
- repository 应优先依赖 dao 与 mapper，而不是自己重新写 SQL
- repository 输出应尽量是稳定的领域对象，而不是上层再去拼 row
- 如果逻辑已经开始关心“启动时扫描插件目录”“迁移外部配置”这类流程，它更适合进 `services/`

判断标准：
- 如果问题是“这个领域对象如何稳定落库 / 读回”，优先放 `repositories/`
- 如果问题是“为了完成一个业务动作，需要协调多个 repository 或外部系统”，继续上移到 `services/`
