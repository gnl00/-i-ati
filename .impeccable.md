## Design Context

### Users
@i is for developers, operators, and power users who use an AI agent as a daily workbench. They manage model providers, MCP servers, skills, plugins, memory, workspace files, tools, schedules, and subagents from one desktop app.

The core job is to stay in a focused conversation while giving the agent enough local context and tool access to act. Chat is the main workspace. Settings, artifacts, task plans, confirmations, and model/tool selectors support the current run.

Users are often in long sessions with streaming output, queued messages, local files, and changing provider state. The interface should make state visible, keep the active turn stable, and keep advanced configuration reachable while preserving conversation focus.

### Brand Personality
The product personality is calm, capable, and technical.

The app should feel like a native macOS AI workbench: quiet chrome, precise controls, and enough personality to make repeated daily use pleasant. The tone is direct and functional, with small moments of warmth in the welcome experience, smart prompts, and motion.

### Aesthetic Direction
The current visual language combines a paper-like chat canvas with translucent desktop surfaces. Primary surfaces use white or near-black glass, soft borders, subtle shadows, and restrained blur. The chat background uses a light/dark dotted paper texture through `bg-chat-light` and `bg-chat-dark`.

UI primitives follow a shadcn/Radix pattern with Tailwind v4 utilities. Use existing components from `src/renderer/src/components/ui` and shared feature controls before introducing new primitives. Prefer lucide or Radix icons inside action buttons.

Layout should be compact and operational. The main chat surface has generous breathing room for messages and the composer, while settings use dense tabs, rows, badges, switches, selectors, and numeric inputs. Cards are appropriate for repeated setting groups, assistant suggestions, tool results, task plans, and contained inspector surfaces.

Color should be mostly neutral with semantic accents. Current accent families include emerald/teal for selected models, amber/orange for MCP and queued state, purple/violet for artifacts, blue/sky for workspace and pinned state, rose/red for destructive or stop states, and green/amber for saved and dirty status.

Motion should be useful and low amplitude. Existing patterns use fade, opacity, small scale, slight translate, shine, tab indicator movement, popover fade/scale, welcome depth, and task/status entrance transitions. Hover states should feel tactile through color, border, shadow, opacity, and small scale changes.

Typography should stay system-native with Inter and platform fallbacks. Hero-scale type belongs in the welcome screen. Settings, tool surfaces, badges, and toolbar controls should use compact sizes, medium weights, and clear hierarchy.

### Design Principles
1. Keep chat as the center of gravity. New UI should support the active conversation, current run state, and workspace context.
2. Preserve native desktop calm. Use translucent surfaces, soft borders, stable spacing, and macOS-friendly controls.
3. Make state visible at the control. Selected models, active MCP servers, artifacts, dirty settings, queued messages, and run status should be visible exactly where the user acts.
4. Reuse established primitives. Build with Radix/shadcn-style components, existing shared selectors, existing tabs, and existing icon-button patterns.
5. Use accents semantically. Keep the neutral base dominant and reserve color for active state, status, category, or risk.
6. Keep motion grounded. Use small, intentional motion for entrance, selection, progress, hover, and focus feedback.
7. Protect long-session readability. Text should truncate or wrap intentionally, controls should keep stable dimensions, and streaming content should maintain scroll position predictably.

### Implementation Notes
- Follow Tailwind v4 syntax in `docs/spec/tailwindcss/tailwindcss-v4-syntax-rules.md` for new Chat UI work.
- Keep app chrome compatible with Electron drag regions through `app-dragable` and `app-undragable`.
- Match the existing settings pattern: animated tabs, icon labels, section cards, badges, right-aligned controls, and explicit dirty/saved state.
- Settings pages should follow the compact control-surface language in `docs/ui/settings-design-language.md`: shared shells, section stacks for independent controls, resource-list panels for collections, and master-detail layouts for provider-style workflows.
- Settings should read as one connected workspace: title, save state, tab bar, and active tab content share a single outer frame, with cards and panels used as internal organization.
- Settings popover hosts should keep a single outer frame while letting shell and content primitives inherit the available popover size (`w-[95vw] h-[93vh]`).
- Providers keep their established selected-state accent: the active provider uses the blue gradient, bottom accent line, icon scale feedback, and expressive delete-hover motion while the surrounding sidebar shell and form controls use settings primitives.
- Match the existing composer pattern: compact pill selectors, icon actions, tooltip-backed buttons, active-state gradients, and subtle `active:scale-*` feedback.
- Use fixed or constrained dimensions for toolbars, icon buttons, selectors, and status chips so state changes keep the layout stable.
